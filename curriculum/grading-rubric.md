## 二、详细评分细则 | Detailed Criteria

### 一、复刻完成度 (30 分) | Replication Completion

考察视频 `BV1tb4y1U7Du` 的核心功能复刻了多少。复刻模式代码已提供，重点在「能不能跑通、跑稳」。

| 分数 | 标准 |
|------|------|
| **27-30** | 全部核心功能实现并稳定：WiFi 联网 + SNTP 对时 + 时间/温湿度显示 + 蜂鸣器报时 + 按键调时 + 断网回退，连续运行 30 分钟无故障 |
| **21-26** | 核心功能实现（时间+温湿度+WiFi 对时），1-2 个完善项（蜂鸣器/按键/断网回退）有小问题但不影响主功能 |
| **15-20** | 基本功能实现（时间+温湿度显示），WiFi 或对时不稳定，需手动干预 |
| **8-14** | 部分功能实现，OLED 或 SHT31 或 WiFi 中有一项不通 |
| **0-7** | 无法运行或只点亮 LED |

**功能检查清单（对照验收）**:
- [ ] OLED 显示时间（网络同步，和手机一致 <2 秒误差）
- [ ] OLED 显示温湿度（SHT31，手捂会变）
- [ ] ESP8266 连上 WiFi（`WIFI GOT IP`）
- [ ] SNTP 网络授时（`CIPSNTPTIME` 返回北京时间）
- [ ] 整点报时（蜂鸣器 PWM）
- [ ] 按键手动调时
- [ ] 断网回退 + 恢复自动重连
- [ ] 连续运行 30 分钟稳定

### 二、架构理解 (15 分) | Architecture Understanding

考察对「为什么这么设计」的理解，通过演示讲解和答疑体现。

| 分数 | 标准 |
|------|------|
| **13-15** | 能清晰讲解 STM32L433+ESP8266 两芯片分工、I2C/UART/AT/SNTP 各协议原理、硬件I2C vs 软件I2C 选择理由、断网回退架构；答疑对答如流 |
| **10-12** | 能讲清系统架构和主要协议，个别细节（如 I2C 地址左移、波特率原理）讲不全 |
| **7-9** | 能说出各模块名称和基本作用，但「为什么」讲不清 |
| **4-6** | 只能演示，讲不出架构和原理 |
| **0-3** | 演示都讲不清，明显没理解 |

**必答知识点**:
- 为什么 STM32L433 要配 ESP8266？（L433 无 WiFi，需协处理器）
- I2C 为什么用开漏+上拉？（线与，多设备共享）
- 为什么 OLED 用软件 I2C、SHT31 用硬件 I2C？（移植老库 + 分总线避干扰）
- `printf` 为什么能走串口？（重定向 `__io_putchar`）
- 为什么 30 分钟对时一次而非每秒？（网络重操作，本地 tick 维护）
- 断网为什么时钟不停？（本地 tick 无条件跑，网络只校正）
- WiFi 配置改哪个文件？（`software/src/env_clock/Core/Inc/wifi_config.h`，`config.template.yaml` 只是只读对照）
- OLED 主显示是哪个函数？（`OLED_ShowClock(year,month,day,hour,min,sec,wday,temp,humi,synced)`，`OLED_ShowTime`/`OLED_ShowDate` 是它的子助手）
- **设计权衡题**：为什么 5V→3.3V 用 LDO (AMS1117) 而不是 Buck（降压 DC-DC）？要点见 `curriculum/day-05.md` Q1（低成本 + 低噪声 + 小负载 + ESP-01S 对纹波敏感 → LDO）。

### 三、接线焊接质量 (15 分) | Wiring & Soldering Quality

| 分数 | 标准 |
|------|------|
| **13-15** | 焊点光亮饱满无虚焊无连锡；走线整洁；外壳专业（3D打印/亚克力），OLED/按键/通风孔到位；通电前做过短路检查 |
| **10-12** | 焊接质量良好偶有小瑕疵；外壳完整但工艺一般；布局合理 |
| **7-9** | 焊接基本能跑但有虚焊/连锡痕迹；外壳简陋（纸盒/裸板） |
| **4-6** | 焊接问题多（多处虚焊/连锡），需多次返修才跑；无外壳 |
| **0-3** | 焊接混乱无法运行或持续短路风险 |

**检查要点**:
- 焊点：光亮火山形（好） vs 颗粒发白（虚焊） vs 球形不爬引脚（少锡）
- 有无连锡（相邻焊盘短路）
- 电源走线：ESP-01S 是否独立供电（AMS1117），VCC/GND 总线规范
- 信号线长度：I2C/UART 线短直，无缠绕
- 外壳：OLED 窗口对齐、按键可按、ESP-01S 通风孔、USB 口开口
- 通电前万用表测 VCC-GND 电阻（非 0）

### 四、调试能力 (15 分) | Debugging Ability

通过过程观察 + 周报/复盘里的踩坑记录体现。

| 分数 | 标准 |
|------|------|
| **13-15** | 能独立用串口日志/万用表/断点定位问题；周报/复盘记录了 ≥4 个真实坑及解决过程；会主动加调试手段（printf/LED 指示/I2C 扫描） |
| **10-12** | 能在提示下排查问题；记录了 2-3 个坑；会用基本调试工具 |
| **7-9** | 遇到问题主要靠问老师；记录了 1 个坑；调试工具使用生疏 |
| **4-6** | 遇到问题束手无策或反复乱改；无踩坑记录 |
| **0-3** | 完全不会调试，问题靠别人解决 |

**调试能力证据**:
- 串口 printf 主动加日志定位
- 万用表测电压/通断/短路
- CubeIDE 断点单步调试
- I2C 扫描器找设备地址
- USB-TTL 直连 ESP-01S 隔离问题
- 逻辑分析仪抓 I2C/UART 波形（加分）

## 四、等级评定 | Grade Determination

> 体验档用三档制（达标 / 良好 / 优秀），不照搬大学的 100 分制 + A/B/C/GPA 那套。重点看你「摸过没摸过、能不能说个大概为什么」，不是精确到几分。

| 等级 | 分数 | 描述 |
|------|------|------|
| **优秀** | 90-100 | 全功能复刻 + 理解深 + 演示流畅 + 有自己的扩展或踩坑反思 |
| **良好** | 80-89 | 核心功能全跑通 + 理解清楚（能答上几个为什么）+ 演示顺 |
| **达标** | 60-79 | 核心功能完成（时间+温湿度+对时能跑）+ 基本理解 + 能演示出来 |
| **未达标** | 0-59 | 核心功能没跑起来，或演示讲不清自己在做什么 |

## 六、评分原则 | Grading Principles

1. **复刻不是减分项**: 固件基于 UP主 代码是本项目的设定，不因「不是从零写」而扣分。评分看的是**理解、整合、调试、工程素养**。
2. **过程重于结果**: 一个功能没跑通但周报/复盘记录了详细排查过程，比跑通了但说不清为什么的高分。
3. **真实优先**: 周报/复盘里的坑必须是真踩过的，抄来的套话扣分。Git 历史能佐证。
4. **鲁棒性加分**: 断网回退、错误处理、稳定性是区分「能跑」和「产品级」的关键，做好了加分。
5. **诚信底线**: 冒充从零原创、抄袭他人代码/报告、Git 造假 → 该维度直接 0 分。
