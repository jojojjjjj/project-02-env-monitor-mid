## 二、全功能回归测试 —— 演示前必做 | Full Regression Test

### 2.1 为什么要回归测试

演示当天翻车是最大噩梦。昨天好好的，今天演示时 WiFi 连不上、OLED 不亮、蜂鸣器不响——这种事在硬件项目里太常见（运输震动、环境 WiFi 不同、电源变化）。**今天上午全部测一遍**，问题当场修。

> **WHY regression test before demo?** 硬件状态会变：运输让接线松了、演示现场 WiFi 和你家的不同、USB 口供电能力不同。回归测试把「能跑」从「我家能跑」变成「任何环境能跑」。这是工程师对演示负责的基本动作。

### 2.2 回归测试清单（逐项打勾）

**A. 上电启动**
- [ ] USB 上电后 <3 秒 OLED 亮起，显示启动画面
- [ ] 串口打印启动日志，无 `FAIL` / `ERROR`
- [ ] 板载 LED 心跳（如有）

**B. WiFi + SNTP**
- [ ] 连上演示现场 WiFi（提前问 SSID/密码，或用手机热点 2.4G）
- [ ] 30 秒内 `WIFI GOT IP`
- [ ] 60 秒内 SNTP 对上时，OLED 显示正确北京时间
- [ ] OLED 显示「WiFi OK」

**C. 时间显示**
- [ ] 时间和手机一致（误差 <2 秒）
- [ ] 秒数每秒在涨，跨分钟/小时正确进位
- [ ] 日期、星期正确

**D. 温湿度**
- [ ] 温度在室温合理范围（18-32°C）
- [ ] 湿度合理（20-80%）
- [ ] 手捂传感器，温度上升

**E. 蜂鸣器**
- [ ] 手动触发整点报时，蜂鸣器响
- [ ] 音量可接受（别太吵影响别人）

**F. 按键**
- [ ] 进入调时模式，能调时/分
- [ ] 退出调时模式，时间继续走

**G. 断网回退**
- [ ] 关掉手机热点 / 拔路由器，OLED 显示「WiFi--」
- [ ] 时钟继续走，不停
- [ ] 恢复网络，60 秒内自动重连重对时

**H. 长时间稳定性**
- [ ] 连续运行 30 分钟无崩溃、无重启、无乱码
- [ ] ESP-01S 和 LDO 温度正常（温热不烫手）

### 2.3 演示现场 WiFi 预案（讲 WHY）

演示现场 WiFi 可能：加密、要认证、5GHz only、信号弱。**必须有预案**：
- **首选**: 提前问现场 WiFi，现场先连好
- **备用 1**: 自己手机开 2.4G 热点（记好热点名/密码，写进 `config.h`）
- **备用 2**: 如果完全没网，演示断网回退——手动调时后展示「时钟照走」，解释「有网自动校准，无网继续运行」

> **WHY a WiFi backup plan?** 网络时钟的演示依赖网络。如果现场网络不行，没预案就只能干瞪眼。准备手机热点是最稳的——你的热点你做主。这也展示了你对「产品在任意环境运行」的工程考虑。

### 3.2 演示要点（讲 WHY）

**实物演示是核心**——评委想看到真东西在动。顺序：
1. 先展示 OLED 显示时间温湿度（最直观）
2. 手捂传感器，温度上升（证明真在读）
3. 触发整点报时，蜂鸣器响（证明 PWM）
4. **断网回退演示**（亮点）：关热点 → 时钟照走 → 开热点 → 自动重连——这个最能体现工程鲁棒性，一定要演

**架构讲解用 Day 1 的系统架构图**，讲清「STM32L433 大脑 + ESP8266 网卡」的分工。这是你理解最深的地方，讲清楚加分。

**复刻历程讲真实踩坑**：
- ESP-01S 供电不足反复复位 → 加独立 AMS1117
- I2C 地址 0x44 写成 0x88（左移 1 位）
- 串口 printf 没重定向、波特率不匹配收乱码
- 这些真实问题比「一切顺利」有说服力，证明你真做过。

### 3.4 答疑预演（讲 WHY）

评委可能问的问题，提前想答案：

| 问题 | 准备的答案要点 |
|------|---------------|
| 为什么用 STM32L433 而不是 ESP32（一片带 WiFi）？ | 复刻视频原设计；学习「主控+协处理器」架构；L433 低功耗 |
| 断网怎么办？ | 本地 tick 继续走，恢复后自动重连重对时（演示过） |
| 精度多少？ | SNTP 毫秒级，本地 tick 30 分钟内漂移 <2 秒 |
| 能扩展什么？ | 加 RTC 断电走时、加 web 配置、加 MQTT 上传数据 |
| 代码是你写的吗？ | 复刻模式：UP主固件为基础，我做了接线/配置/调试/整合/优化 |
| I2C 为什么用两套（硬件+软件）？ | SHT31 硬件 I2C2 准，OLED 软件 I2C 移植老库且分总线避干扰 |

### 5.3 给下一个项目积累的经验（讲 WHY）

把这次踩的坑抽象成可复用的经验：

1. **供电永远是第一个怀疑对象**：模块不稳先查电流够不够，加电容，独立 LDO
2. **协议要确认两端参数一致**：波特率、I2C 地址、数据格式——不一致 = 乱码/无响应
3. **中断接收 + 环形缓冲区**：任何「不定长异步数据」都用这个模式
4. **本地维护 + 网络校正分离**：网络依赖要降级容错，核心功能本地能跑
5. **边焊边测、通电前查短路**：硬件调试的铁律

> **WHY abstract lessons?** 具体的「ESP-01S 供电」只对这个项目有用，但「供电永远先怀疑」适用于所有硬件项目。把经验从「具体」抽象到「通用」，你才真正在成长——这是从「做完一个项目」到「成为工程师」的关键一步。

## 八、调试技巧（最后一组）| Final Debugging Tips

1. **演示前别动代码**: 今天上午回归测试发现问题，只修 bug 不加新功能。加新功能 = 引入新 bug = 演示翻车。
2. **带备份**: USB 线 2 根、备用 ESP-01S 1 个、备用 KE1 板（如果有）。关键元件有备无患。
3. **录屏存证**: 演示前用手机录一段完整功能视频——万一现场翻车，放视频也算演示。
4. **串口日志存档**: `assignments/` 里存一份正常运行时的串口日志文本，证明功能跑过。

## 十、结业 | Closing

恭喜完成 Project 02。10 天前你可能没碰过 STM32，现在你有了一台自己焊、自己调、能联网对时的桌面温湿度网络时钟。

这 10 天你掌握的不只是这个项目——你掌握了**嵌入式硬件工程的标准流程**：看架构 → 选器件 → 接线 → 配工具链 → 烧固件 → 看串口 → 排错 → 焊接 → 组装 → 演示。这套流程适用于你以后做的任何硬件项目：智能手表、机器人、智能家居、 whatever。

> The skills you learned here — toolchain, protocols, debugging, soldering, system integration, robustness — transfer to every hardware project you'll ever build. This clock is your first real embedded product. Make more.

下一步可以探索的方向：
- **PCB 进阶**: 用立创 EDA 画定制 PCB，发嘉立创打样
- **加 RTC**: DS3231 硬件 RTC + 纽扣电池，断电也能走时
- **上云**: 加 MQTT 把温湿度上传到 EMQX/阿里云 IoT，做数据可视化
- **低功耗**: STM32L433 的 Stop 模式 + 唤醒，电池供电
- **屏幕升级**: 换 1.3" OLED 或 SPI TFT 彩屏，显示更丰富

带着这 10 天的经验，去更难的项目（Project 03 键盘、Project 07 机器人）继续打怪升级。
