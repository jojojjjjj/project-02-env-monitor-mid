## 二、全功能回归测试 —— 演示前必做 | Full Regression Test

### 2.1 为什么要回归测试

演示当天翻车是最大噩梦。昨天好好的，今天演示时 WiFi 连不上、OLED 不亮、蜂鸣器不响——这种事在硬件项目里太常见（运输震动、环境 WiFi 不同、电源变化）。**今天上午全部测一遍**，问题当场修。

> **WHY regression test before demo?** 硬件状态会变：运输让接线松了、演示现场 WiFi 和你家的不同、USB 口供电能力不同。回归测试把「能跑」从「我家能跑」变成「任何环境能跑」。这是工程师对演示负责的基本动作。

### 2.2 回归测试清单（逐项打勾）⭐⭐

**A. 上电启动**
- [ ] USB 上电后 <3 秒 OLED 亮起，显示启动画面
- [ ] 串口打印启动日志，无 `FAIL` / `ERROR`
- [ ] 板载 LED 心跳（如有）

**B. WiFi + SNTP**
- [ ] 连上演示现场 WiFi（提前问 SSID/密码，或用手机热点 2.4G）
- [ ] 30 秒内 `WIFI GOT IP`
- [ ] 60 秒内 SNTP 对上时，OLED 显示正确北京时间
- [ ] OLED 显示「WiFi OK」

**C. 时间显示**
- [ ] 时间和手机一致（误差 <2 秒）
- [ ] 秒数每秒在涨，跨分钟/小时正确进位
- [ ] 日期、星期正确

**D. 温湿度**
- [ ] 温度在室温合理范围（18-32°C）
- [ ] 湿度合理（20-80%）
- [ ] 手捂传感器，温度上升

**E. 蜂鸣器**
- [ ] 手动触发整点报时，蜂鸣器响
- [ ] 音量可接受（别太吵影响别人）

**F. 按键**
- [ ] 进入调时模式，能调时/分
- [ ] 退出调时模式，时间继续走

**G. 断网回退**
- [ ] 关掉手机热点 / 拔路由器，OLED 显示「WiFi--」
- [ ] 时钟继续走，不停
- [ ] 恢复网络，60 秒内自动重连重对时

**H. 长时间稳定性**
- [ ] 连续运行 30 分钟无崩溃、无重启、无乱码
- [ ] ESP-01S 和 LDO 温度正常（温热不烫手）

### 2.3 演示现场 WiFi 预案（讲 WHY）⭐⭐

演示现场 WiFi 可能：加密、要认证、5GHz only、信号弱。**必须有预案**：
- **首选**: 提前问现场 WiFi，现场先连好
- **备用 1**: 自己手机开 2.4G 热点（记好热点名/密码，写进 `software/src/env_clock/Core/Inc/wifi_config.h`）
- **备用 2**: 如果完全没网，演示断网回退——手动调时后展示「时钟照走」，解释「有网自动校准，无网继续运行」

> **WHY a WiFi backup plan?** 网络时钟的演示依赖网络。如果现场网络不行，没预案就只能干瞪眼。准备手机热点是最稳的——你的热点你做主。这也展示了你对「产品在任意环境运行」的工程考虑。

### 5.3 给下一个项目积累的经验（讲 WHY）⭐⭐

## 理解验证问题 | Comprehension Check

> 这些是评委大概率会问的「为什么」。提前想清楚，演示答疑才不慌。先自己想，再翻参考答案。

**Q1（原理追问题）** 为什么这个项目用「STM32L433 主控 + ESP8266 协处理器」两片芯片，而不是直接用一片带 WiFi 的 ESP32？

> **参考答案**: 三个角度——①**复刻忠实度**：本课程复刻 UP 主「乐在程上」的 KE1 原设计，原设计就是 STM32 + ESP8266 架构，保持一致便于对照学习。②**架构学习价值**：两片芯片能让你亲身理解「主控 + 协处理器」的分工——主控管本地逻辑（读传感器、刷显示、走时），协处理器专门管联网（WiFi + SNTP），两者通过 UART + AT 命令通信。这是物联网设备的经典架构（比如很多智能家电都是 MCU + WiFi 模组）。一片 ESP32 把所有事包了，反而学不到这种分工。③**STM32L433 的低功耗**：L4 系列是 ST 的低功耗线，适合做电池供电的摆件；ESP32 功耗高、且其 GPIO 驱动/外设生态和 STM32 不同。代价是多一片芯片、多一套 UART 通信、AT 命令框架要自己调——但对学习「工程架构」反而更值。

**Q2（故障分析题）** 演示现场 WiFi 连不上，你有哪几道备用方案？按「最稳 → 最险」排序，并说明每道方案适合什么情况。

> **参考答案**: ①**首选：提前问现场 WiFi，现场先连好**——最稳，但依赖现场配合、可能有认证墙。②**备用 1：自己手机开 2.4G 热点**（热点名/密码提前写进 `wifi_config.h`）——很稳，你的热点你做主，唯一要求是手机电量/流量够。③**备用 2：演示断网回退**（完全没网时）——手动按键调时后展示「时钟照走」，解释「有网自动校准，无网继续运行」。这不算翻车，反而展示了工程鲁棒性，但「联网对时」这个卖点演示不了，是最后手段。排序：① > ② > ③。

**Q3（设计权衡题 — A 而不是 B）** 断网回退功能里，为什么要「本地 tick 继续走」而不是「断网就停表等重连」？

> **参考答案**: 时钟的核心价值是「显示当前时间」。断网就停表，用户看到一个冻住的时间——既不准又不可信，体验极差。本地 tick（用 STM32 的 SysTick/RTC 维护时间）让时钟在断网时继续走，虽然会慢慢漂移（本地晶振精度有限，30 分钟可能漂 1-2 秒），但用户看到的是「还在走、大致准」的时钟。等网络恢复，SNTP 再把它校准回精准时间。这就是**「核心功能本地能跑、网络依赖降级容错」**的设计原则——网络是加分项不是命门。类比：手机开飞行模式，时钟照走、闹钟照响，不会因为没网就停摆。
