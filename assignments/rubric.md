## 📊 一、总体评分分布 | Overall Grading Distribution

本项目是**复刻模型**（firmware 代码已提供，学生做接线+烧录+配置+调试，不从零写代码），因此评分侧重**完成度、理解、动手与调试**，而非代码原创性。

| 类别 (Category) | 权重 | 满分 | 说明 |
|----------------|:----:|:----:|------|
| **复刻完成度** (Replication Completion) | 30% | 30 分 | 时钟各项功能是否实现并稳定运行 |
| **架构理解** (Architecture Understanding) | 15% | 15 分 | 能否讲清四大模块分工与设计取舍 |
| **接线焊接质量** (Wiring & Soldering) | 15% | 15 分 | 接线正确、整洁、焊接可靠 |
| **调试能力** (Debugging Ability) | 15% | 15 分 | 遇到问题能否定位根因并解决 |
| **演示** (Demo Presentation) | 15% | 15 分 | 现场演示流畅、讲解清晰 |
| **文档** (Documentation) | 10% | 10 分 | 周报、README、注释、Git 记录 |
| **基础总分** | **100%** | **100 分** | |
| 额外加分 (Bonus) | +10% | +10 分 | 创新亮点、开源贡献 |
| **最高总分** | | **110 分** | |

## 🎯 二、详细评分细则 | Detailed Scoring Criteria

### 一、复刻完成度 (30 分)

> 验收的是**整机功能**，不是单模块。每项要能现场演示。

#### 功能检查清单 (Function Checklist)

| 功能项 | 分值 | 验收标准 |
|--------|:----:|---------|
| STM32 基础运行 | 3 分 | 板子能启动、烧录、串口 printf 正常 |
| SHT31 温湿度读取 | 4 分 | 能读到合理温湿度（手捂会变），数据稳定 |
| OLED 显示 | 4 分 | SSD1306 正常显示字符与温湿度 |
| WiFi 联网 | 5 分 | ESP8266 AT 连上 WiFi，拿到 IP |
| SNTP 网络授时 | 6 分 | 同步到网络时间，与标准时间误差 <2 秒 |
| 时间 + 温湿度同屏 | 4 分 | OLED 同时显示 HH:MM:SS 和温湿度，布局清晰 |
| 整点报时 | 2 分 | 到整点蜂鸣器响一声 |
| 断网回退 + 重连 | 2 分 | 断网后时钟继续走，恢复后自动重新校时 |

**完成度评分等级:**

| 分数段 | 等级 | 标准 |
|--------|------|------|
| 27–30 | 优秀 | 所有功能实现并稳定运行 ≥10 分钟无崩溃 |
| 21–26 | 良好 | 核心功能（联网+授时+显示）实现，有少量瑕疵 |
| 15–20 | 及格 | 基本功能实现，但缺整点报时/断网回退等进阶项 |
| 0–14 | 不及格 | 核心功能不完整或无法稳定运行 |

**扣分项:**

- 核心功能缺失（无 WiFi 联网 / 无授时 / 无显示）：-5 分/项
- 功能不稳定（演示中崩溃 / 数据跳变）：-2 分/次
- 数据错误（温湿度明显异常 / 时间偏移大）：-2 分/项

---

### 二、架构理解 (15 分)

> 复刻不是照葫芦画瓢。学生要能讲清“为什么这样设计”。

#### 理解检查点 (Understanding Checkpoints)

| 检查点 | 分值 | 评估要点 |
|--------|:----:|---------|
| 主控 + 协处理器分工 | 4 分 | 能讲清为什么 STM32L433 当主控、ESP8266 当 WiFi 协处理器 |
| 通信总线 | 4 分 | 能讲清 I2C（SHT31/OLED）、UART/AT（ESP8266）、PWM（蜂鸣器）各自的角色 |
| SNTP 授时原理 | 3 分 | 能讲清为什么用 SNTP、时区、与本地 RTC 的取舍 |
| 数据流 | 2 分 | 能画出“ESP8266→SNTP→STM32→OLED”的数据流 |
| 设计取舍 | 2 分 | 能说出至少一个“为什么这么选”的工程权衡 |

**理解评分等级:**

| 分数段 | 等级 | 标准 |
|--------|------|------|
| 13–15 | 优秀 | 能清晰讲解全部架构，对设计取舍有深入理解 |
| 10–12 | 良好 | 能讲清主要模块分工，个别细节说不透 |
| 7–9 | 及格 | 能说出模块名称，但讲不清“为什么” |
| 0–6 | 不及格 | 架构概念混乱，答非所问 |

**典型问题 (用于口头考查):**

- 为什么用 ESP8266 而不直接用带 WiFi 的 MCU？
- SHT31 和 OLED 都用 I2C，为什么不会冲突？
- 断网后时钟靠什么继续走？精度如何？
- 为什么 SNTP 时间比手机晚 8 小时？（时区）

---

### 三、接线焊接质量 (15 分)

> 硬件动手能力是复刻项目的核心产出之一。

#### 质量检查项 (Quality Checkpoints)

| 检查项 | 分值 | 标准 |
|--------|:----:|------|
| 接线正确性 | 5 分 | 所有引脚对照接线表无误（电源 3.3V/5V 不混淆、TX/RX 不接反、SCL/SDA 不接反） |
| 接线整洁度 | 3 分 | 走线整齐、用扎带/胶带固定、可整体移动不松脱 |
| 焊接质量（PCB 版） | 5 分 | 焊点光亮饱满、无虚焊连锡、元件方向正确 |
| 电源可靠性 | 2 分 | 供电稳定，无因供电不足导致复位/异常 |

> 注：选面包板 MVP 方案的学生，"焊接质量"5 分按"接线牢固度与可维护性"评估（杜邦线接触是否可靠、是否易松动）。

**质量评分等级:**

| 分数段 | 等级 | 标准 |
|--------|------|------|
| 13–15 | 优秀 | 接线正确整洁、焊接专业、整机可靠可移动 |
| 10–12 | 良好 | 接线正确，整洁度一般，焊接基本可靠 |
| 7–9 | 及格 | 接线有 1–2 处需返工，焊接有少量虚焊 |
| 0–6 | 不及格 | 接线多处错误、松散混乱、焊接问题导致功能异常 |

---

### 四、调试能力 (15 分)

> 看学生在周报“问题记录表”里体现的**根因分析**深度，以及现场答疑时解决问题的思路。

#### 调试能力检查项 (Debugging Checkpoints)

| 检查项 | 分值 | 标准 |
|--------|:----:|------|
| 问题记录完整性 | 4 分 | 周报里记录了真实卡点，含现象+根因+解决方案 |
| 根因分析深度 | 5 分 | 能定位到根本原因（不只是“重启就好了”），讲清 why |
| 工具使用 | 3 分 | 会用串口助手、万用表、逻辑分析仪（可选）辅助调试 |
| 独立解决 | 3 分 | 主要问题自己解决，而非全程依赖导师 |

**调试评分等级:**

| 分数段 | 等级 | 标准 |
|--------|------|------|
| 13–15 | 优秀 | 多个真实卡点，根因分析深入，独立解决为主 |
| 10–12 | 良好 | 记录了问题，根因基本讲清，部分靠提示 |
| 7–9 | 及格 | 有问题记录但根因分析浅（只到现象层） |
| 0–6 | 不及格 | 无问题记录或全靠他人代劳 |

### 五、演示 (15 分)

> 现场演示 + 讲解。见 `final-presentation.md`。

#### 演示检查项 (Demo Checkpoints)

| 检查项 | 分值 | 标准 |
|--------|:----:|------|
| 功能现场演示 | 6 分 | 联网+授时+显示+报时+断网回退，现场跑通 |
| 讲解清晰度 | 4 分 | 架构、流程、挑战讲得清楚，逻辑顺畅 |
| 时间控制 | 2 分 | 5–8 分钟内完成，不超时不少时 |
| 问答应对 | 3 分 | 回答问题准确，能承认不足 |

**演示评分等级:**

| 分数段 | 等级 | 标准 |
|--------|------|------|
| 13–15 | 优秀 | 演示流畅无卡顿，讲解深入，问答从容 |
| 10–12 | 良好 | 演示基本顺利，讲解清楚，问答基本应对 |
| 7–9 | 及格 | 演示有小问题，讲解一般，问答勉强 |
| 0–6 | 不及格 | 演示失败或严重超时，讲解混乱 |

### 六、文档 (10 分)

#### 文档检查项 (Documentation Checkpoints)

| 文档 | 分值 | 标准 |
|------|:----:|------|
| Week 1 / Week 2 周报 | 4 分 | 按时提交、内容完整、问题分析有深度（见 `week-1-checkin.md` / `week-2-checkin.md`） |
| README | 2 分 | 项目说明完整：是什么、怎么编译烧录、怎么配置 WiFi、中英双语 |
| Git 提交记录 | 3 分 | 用 Conventional Commits 规范、提交频率合理、已推送 GitHub |
| 代码注释 | 1 分 | 关键逻辑（AT 命令序列、时间解析、整点报时）有注释 |

**Git 提交规范 (Git Commit Convention):**

```bash
# 好的提交 (规范)
git commit -m "feat(sht31): SHT31 驱动初始化 + 读温湿度成功"
git commit -m "feat(sntp): SNTP 授时解析，本地时间同步"
git commit -m "fix(clock): 断网后时钟卡死，改用 SysTick 本地计时"
git commit -m "build(toolchain): 导入 KE1 参考工程，编译通过"
git commit -m "docs: 填写 week-1-checkin.md"

# 不好的提交 (不规范，扣分)
git commit -m "update"
git commit -m "fix bug"
git commit -m "..."
```

## 📈 三、等级评定 | Grade Determination

> 体验档用三档制（达标 / 良好 / 优秀），不照搬大学 100 分制 + A/B/C/GPA 那套。重点看你「摸过没摸过、能不能说个大概为什么」，不是精确到几分。

| 等级 | 分数范围 | 描述 |
|------|---------|------|
| **优秀** | 90–100 | 全部功能实现并稳定，理解深入，调试独立，有自己的扩展或反思 |
| **良好** | 80–89 | 核心功能扎实，进阶项部分完成，能答上几个「为什么」 |
| **达标** | 60–79 | 核心功能跑通（时间+温湿度+对时），基本理解，能演示出来 |
| **未达标** | 0–59 | 核心功能没跑起来，或演示讲不清自己在做什么 |

## 🌟 四、额外加分项 | Bonus Points

> 基础总分 100 之外，最多加 10 分。

### 创新亮点 (最多 +5 分)

| 创新类型 | 加分 | 示例 |
|---------|:----:|------|
| 功能创新 | +3 分 | 实现网页配 WiFi、天气图标、温湿度曲线图等计划外功能 |
| 性能优化 | +2 分 | 显著降低功耗、优化刷新时序 |
| 外观设计 | +2 分 | 3D 打印外壳 / 精致 PCB / 独特 UI 布局 |
| 复刻超越 | +2 分 | 在原视频基础上增加有价值的新特性 |

### 开源贡献 (最多 +3 分)

| 贡献类型 | 加分 |
|---------|:----:|
| 代码开源到 GitHub（含清晰 README） | +2 分 |
| 编写详细复刻教程 / 踩坑笔记 | +1 分 |
| 帮助其他同学解决问题 | +1 分 |

### 社区参与 (最多 +2 分)

| 参与方式 | 加分 |
|---------|:----:|
| 在 Bilibili 视频下分享复刻心得 | +1 分 |
| 撰写技术博客 | +1 分 |

**最高总分：110 分（含加分）**
